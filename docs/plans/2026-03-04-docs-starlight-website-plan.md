# Plan: Add Docs Website (Starlight + Bun)

## Context

Roombarr is a NestJS backend project running on Bun 1.3.6. The `docs/` directory is reserved for internal development docs generated by the Claude Code compound engineering plugin. We need a user-facing documentation site under `website/` deployed to GitHub Pages.

**Scope:** Minimal scaffolding only — one placeholder page, build pipeline, deployment workflow. No content, styling, or plugin work.

## Decision: Starlight (Astro)

**Why Starlight over alternatives:**

- **Bun compatibility:** Astro/Starlight works with Bun for `install`, `dev`, and `build` in static output mode (which is the default and exactly what we need). No second runtime required.
- **Static-only use case:** Starlight defaults to static site generation — no SSR, no MDX required for basic setup (`.md` files work natively). This avoids the known Bun + MDX incompatibility.
- **GitHub Pages:** Astro has a first-party `withastro/action` for GitHub Pages deployment, and we can alternatively use a manual Bun-based workflow for consistency with the rest of CI.
- **Ecosystem:** Starlight is the most actively maintained Astro docs theme with 735+ code snippets in official docs, high reputation, and a large community.

**Rejected alternatives:**

- **Docusaurus:** Requires Node.js — would force managing two runtimes.
- **VitePress:** Vue-based, less natural fit. Also Node-oriented.
- **MkDocs:** Python-based — even further from the project's toolchain.

## Steps

### 1. Scaffold Starlight under `website/`

Create a minimal Starlight project structure manually (not via `create astro` which targets the root). Files:

- `website/package.json` — Astro + Starlight dependencies, scripts (`dev`, `build`, `preview`)
- `website/astro.config.mjs` — Starlight config with `site` set to `https://roombarr.github.io/roombarr/` and `base` set to `/roombarr/`
- `website/tsconfig.json` — Extends Astro's strict preset
- `website/src/content/docs/index.md` — Single landing page with project name and one-liner description (`.md`, not `.mdx` — avoids MDX integration and its known Bun incompatibility)
- `website/src/content.config.ts` — Starlight content collection config (required by Astro content layer)

### 2. Configure root project integration

- **`biome.json`:** Add `website/` to the `files.ignore` list — Starlight uses its own config conventions and the root Biome rules (e.g., `noDefaultExport: error`) would conflict with Astro's config files and page components.
- **`.gitignore`:** Add `website/.astro/` entry. (`node_modules/` and `dist/` are already ignored by existing root-level glob rules that match all depths.)
- **Root `package.json`:** Add convenience scripts: `docs:dev`, `docs:build`, `docs:preview` that run `bun --cwd website <command>`.
- **`lint-staged.config.ts`:** No changes needed — it delegates to Biome, which will skip `website/` once the ignore is in place.

### 3. Add GitHub Actions workflow

Create `.github/workflows/deploy-docs.yml`:

```yaml
name: Deploy Docs

on:
  push:
    branches: [main]
    paths: ["website/**"]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
      - run: bun install --frozen-lockfile
        working-directory: website
      - run: bun run build
        working-directory: website
      - uses: actions/upload-pages-artifact@v3
        with:
          path: website/dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

**Design choices:**

- Uses `oven-sh/setup-bun@v2` instead of `withastro/action` — keeps Bun as the sole runtime in CI, consistent with existing workflows.
- `paths: ['website/**']` filter — only triggers on docs changes, not on every push to main.
- `workflow_dispatch` — allows manual deployment.
- `concurrency` group prevents parallel deploys.
- Separate `bun.lock` in `website/` keeps the docs dependency tree isolated from the main app.

### 4. Verify locally

- Run `bun install` in `website/`
- Run `bun run dev` — confirm the dev server starts and the landing page renders
- Run `bun run build` — confirm static output generates to `website/dist/`
