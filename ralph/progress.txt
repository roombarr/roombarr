## Codebase Patterns
- Controller at src/evaluation/evaluation.controller.ts explicitly picks serialized fields — `EvaluationRun` has more fields than the API exposes
- `execution_status` and `execution_error` are optional (`?`) on `EvaluationItemResult` in src/rules/types.ts
- `keep` actions are skipped before execution counting in src/execution/action-executor.service.ts (line 54)
- Docs site lives under `website/` not `docs/`
- Typecheck: `npm run typecheck`; docs build: `npm run docs:build`
- lint-staged runs biome on commit
- Operator functions have varying arities: `is_empty`/`is_not_empty` only use field arg; `includes_all` guards both field AND value with `Array.isArray()`
- Field resolver (src/rules/field-resolver.ts) uses dynamic dot-notation traversal — fields exist at runtime even if not in field-registry.ts, but config validation rejects unregistered fields
- Operator compatibility table canonical home is reference/fields.md (old anchor was #type-to-operator-compatibility, now #operator-compatibility)
- State tracking system is generic (supports `targets: Array<'radarr' | 'sonarr'>`) but all current state fields track Radarr-specific data
- `schedule` field is `z.string().min(1)` — no cron validation at config load; invalid cron fails at runtime
- `name` field on rules is `z.string().min(1)` — no uniqueness constraint
- Leaf condition `field` path is validated with regex `/^[a-z][a-z0-9_.]*$/` — must start with lowercase letter
